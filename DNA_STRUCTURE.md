# Estructura del ADN Organizacional - SNISSI\n\n## Introducción\n\nEl ADN Organizacional es el repositorio centralizado de conocimiento que guía el funcionamiento del sistema SNISSI. Contiene reglas, políticas, criterios, plantillas y aprendizajes acumulados.\n\n## Componentes del ADN\n\n### 1. POLICY (Políticas)\n\n**Propósito**: Directrices generales que rigen la organización.\n\n**Características**:\n- Nivel más alto de abstracción\n- Aplican a múltiples procesos\n- Cambian infrecuentemente\n- Requieren aprobación ejecutiva\n\n**Ejemplos**:\n```\nKey: SALARY_POLICY_2026\nValue: \"Incremento salarial anual del 5% para todos los empleados con evaluación positiva\"\n\nKey: BENEFIT_POLICY_SENIOR_STAFF\nValue: \"Empleados con más de 10 años reciben 2 días adicionales de vacaciones\"\n\nKey: OVERTIME_POLICY\nValue: \"Horas extra se pagan con factor 1.25x en días laborales, 1.5x en fines de semana\"\n```\n\n### 2. RULE (Reglas)\n\n**Propósito**: Normas específicas para cálculos y decisiones.\n\n**Características**:\n- Más específicas que políticas\n- Implementan políticas\n- Pueden cambiar según contexto\n- Guían lógica de agentes\n\n**Ejemplos**:\n```\nKey: SALARY_CALCULATION_METHOD\nValue: \"Salario = Salario Base + Bonificación + Horas Extra - Descuentos - Impuestos\"\n\nKey: BONUS_CALCULATION_RULE\nValue: \"Bonificación = (Salario Base * 0.15) si evaluación >= 4/5, (Salario Base * 0.10) si evaluación >= 3/5\"\n\nKey: DEDUCTION_PRIORITY\nValue: \"Orden de descuentos: 1) Impuestos, 2) Seguridad Social, 3) Préstamos, 4) Otros\"\n```\n\n### 3. CRITERION (Criterios)\n\n**Propósito**: Estándares de validación y evaluación.\n\n**Características**:\n- Definen umbrales y límites\n- Usados para detección de anomalías\n- Cuantitativos o cualitativos\n- Pueden variar por categoría de empleado\n\n**Ejemplos**:\n```\nKey: SALARY_ANOMALY_THRESHOLD\nValue: \"Variación de más del 20% respecto al mes anterior es anomalía\"\n\nKey: OVERTIME_LIMIT_MONTHLY\nValue: \"Máximo 40 horas extra por mes, más de 50 requiere aprobación\"\n\nKey: DEDUCTION_LIMIT_PERCENTAGE\nValue: \"Total de descuentos no puede exceder 50% del salario bruto\"\n\nKey: MISSING_DATA_THRESHOLD\nValue: \"Si faltan más de 3 campos críticos, rechazar liquidación\"\n```\n\n### 4. TEMPLATE (Plantillas)\n\n**Propósito**: Formatos reutilizables para documentos y comunicaciones.\n\n**Características**:\n- Estandarizan presentación\n- Aseguran consistencia\n- Incluyen variables dinámicas\n- Pueden tener múltiples versiones\n\n**Ejemplos**:\n```\nKey: PAYROLL_REPORT_TEMPLATE\nValue: \"\"\"\nREPORTE DE NÓMINA\n================\nPeríodo: {PERIOD}\nCliente: {CLIENT_NAME}\nEmpleados: {EMPLOYEE_COUNT}\n\nRESUMEN FINANCIERO\n- Total Salarios: {TOTAL_SALARY}\n- Total Bonificaciones: {TOTAL_BONUS}\n- Total Descuentos: {TOTAL_DEDUCTIONS}\n- Total Neto: {TOTAL_NET}\n\nPREPARADO POR: {SUPERVISOR_NAME}\nFECHA: {DATE}\n\"\"\"\n\nKey: ANOMALY_ALERT_TEMPLATE\nValue: \"\"\"\nALERTA DE ANOMALÍA\n==================\nCaso: {CASE_ID}\nEmpleado: {EMPLOYEE_NAME}\nTipo de Anomalía: {ANOMALY_TYPE}\nDetalles: {ANOMALY_DETAILS}\n\nACCIÓN REQUERIDA: Revisar y aprobar/rechazar\n\"\"\"\n```\n\n### 5. LEARNING (Aprendizajes)\n\n**Propósito**: Conocimiento adquirido de decisiones humanas.\n\n**Características**:\n- Generados automáticamente de supervisión\n- Representan excepciones o casos especiales\n- Mejoran precisión de agentes\n- Se acumulan con el tiempo\n\n**Ejemplos**:\n```\nKey: OVERTIME_EXCEPTION_SENIOR_STAFF_CASE_123\nValue: \"Para empleado ID 456 (Senior Developer, 15 años antigüedad), \n        horas extra se calculan con factor 1.75x en lugar de 1.25x \n        según acuerdo especial firmado en 2025\"\n\nKey: BONUS_ADJUSTMENT_CASE_456\nValue: \"Empleado ID 789 (Manager), bonificación se calcula sobre \n        salario base + comisiones, no solo salario base\"\n\nKey: DEDUCTION_EXCEPTION_CASE_789\nValue: \"Empleado ID 101 tiene descuento por préstamo que no puede \n        exceder 30% del salario (excepción a regla de 50%)\"\n```\n\n### 6. BEST_PRACTICE (Mejores Prácticas)\n\n**Propósito**: Procesos y enfoques probados.\n\n**Características**:\n- Basadas en experiencia\n- Mejoran eficiencia\n- Reducen errores\n- Documentan conocimiento tácito\n\n**Ejemplos**:\n```\nKey: DOCUMENT_VALIDATION_PROCESS\nValue: \"\"\"\n1. Verificar que todos los documentos requeridos estén presentes\n2. Validar que los datos sean legibles y completos\n3. Comparar con registros anteriores\n4. Flagear inconsistencias para revisión manual\n5. Documentar cualquier cambio o corrección\n\"\"\"\n\nKey: ANOMALY_INVESTIGATION_PROCESS\nValue: \"\"\"\n1. Revisar el tipo de anomalía detectada\n2. Comparar con histórico del empleado\n3. Verificar cambios recientes en datos\n4. Consultar con RH si es necesario\n5. Documentar causa raíz\n6. Registrar como aprendizaje si es patrón\n\"\"\"\n```\n\n## Estructura de Datos\n\n### Tabla: dna_organizational\n\n```sql\nCREATE TABLE dna_organizational (\n  id INT PRIMARY KEY AUTO_INCREMENT,\n  category ENUM('POLICY', 'RULE', 'CRITERION', 'TEMPLATE', 'LEARNING', 'BEST_PRACTICE'),\n  key VARCHAR(255) UNIQUE,\n  value LONGTEXT,\n  description TEXT,\n  version INT DEFAULT 1,\n  active BOOLEAN DEFAULT TRUE,\n  created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,\n  updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP\n);\n```\n\n### Campos\n\n| Campo | Tipo | Descripción |\n|-------|------|-------------|\n| id | INT | Identificador único |\n| category | ENUM | Categoría del elemento |\n| key | VARCHAR(255) | Identificador único del elemento |\n| value | LONGTEXT | Contenido/Descripción |\n| description | TEXT | Notas adicionales |\n| version | INT | Número de versión |\n| active | BOOLEAN | Si está activo o archivado |\n| created_at | TIMESTAMP | Fecha de creación |\n| updated_at | TIMESTAMP | Fecha de última actualización |\n\n## Convenciones de Nomenclatura\n\n### Claves\n\nUse convención SNAKE_CASE:\n- `SALARY_POLICY_2026`\n- `OVERTIME_CALCULATION_RULE`\n- `ANOMALY_THRESHOLD_PERCENTAGE`\n- `PAYROLL_REPORT_TEMPLATE`\n- `BONUS_EXCEPTION_CASE_123`\n\n### Categorización\n\nSiga esta jerarquía:\n1. **POLICY**: Nivel más alto, directrices generales\n2. **RULE**: Implementación de políticas\n3. **CRITERION**: Validación de reglas\n4. **TEMPLATE**: Formatos de salida\n5. **LEARNING**: Excepciones y casos especiales\n6. **BEST_PRACTICE**: Procesos recomendados\n\n## Ciclo de Vida del ADN\n\n### Creación\n\n1. Identificar necesidad (nueva política, regla, etc.)\n2. Documentar en formato estructurado\n3. Obtener aprobación si es necesario\n4. Crear en el sistema\n5. Comunicar a supervisores\n\n### Mantenimiento\n\n1. Revisar regularmente (trimestral)\n2. Actualizar versiones cuando cambia\n3. Archivar elementos obsoletos\n4. Consolidar elementos duplicados\n5. Documentar cambios\n\n### Evolución\n\n1. Supervisores generan aprendizajes\n2. Aprendizajes se agregan automáticamente\n3. Periódicamente revisar aprendizajes\n4. Consolidar en reglas si es patrón\n5. Actualizar políticas si es necesario\n\n## Acceso y Permisos\n\n### Lectura\n- **Todos los usuarios**: Pueden ver ADN\n- **Supervisores**: Pueden ver y usar en decisiones\n- **Administradores**: Acceso completo\n\n### Escritura\n- **Supervisores**: Pueden crear aprendizajes\n- **Administradores**: Pueden crear/editar/eliminar\n\n### Auditoría\n- Todos los cambios se registran\n- Se mantiene historial de versiones\n- Se registra quién hizo cada cambio\n\n## Consulta de ADN\n\n### Por Agentes\n\nLos agentes consultan el ADN:\n- **Interpretador Normativo**: Consulta POLICIES y RULES\n- **Preliquidador**: Consulta RULES, CRITERIA, TEMPLATES\n- **Auditor**: Consulta CRITERIA, LEARNING\n- **Comunicador**: Consulta TEMPLATES, BEST_PRACTICES\n\n### Por Supervisores\n\nLos supervisores pueden:\n- Buscar por categoría\n- Buscar por palabra clave\n- Ver historial de cambios\n- Crear nuevos elementos\n\n## Ejemplos de Consultas\n\n### Obtener todas las políticas\n\n```sql\nSELECT * FROM dna_organizational \nWHERE category = 'POLICY' AND active = TRUE\nORDER BY updated_at DESC;\n```\n\n### Obtener reglas de cálculo\n\n```sql\nSELECT * FROM dna_organizational \nWHERE category = 'RULE' AND key LIKE '%CALCULATION%'\nAND active = TRUE;\n```\n\n### Obtener aprendizajes recientes\n\n```sql\nSELECT * FROM dna_organizational \nWHERE category = 'LEARNING'\nORDER BY created_at DESC\nLIMIT 10;\n```\n\n## Mejores Prácticas para el ADN\n\n1. **Claridad**: Escriba de forma clara y específica\n2. **Completitud**: Incluya todos los detalles relevantes\n3. **Consistencia**: Use convenciones uniformes\n4. **Actualización**: Mantenga el ADN actualizado\n5. **Documentación**: Registre cambios y motivos\n6. **Revisión**: Revise regularmente con el equipo\n7. **Consolidación**: Elimine duplicados y obsoletos\n8. **Versionado**: Mantenga historial de cambios\n\n## Integración con Agentes\n\nCada agente integra el ADN en su lógica:\n\n```typescript\n// Ejemplo: Agente Preliquidador consultando ADN\nconst salaryRules = await getDNAByCategory('RULE');\nconst salaryCalculationRule = salaryRules.find(\n  r => r.key === 'SALARY_CALCULATION_METHOD'\n);\n\nif (salaryCalculationRule) {\n  // Aplicar la regla al cálculo\n  const salary = calculateSalary(employee, salaryCalculationRule.value);\n}\n```\n\n## Conclusión\n\nEl ADN Organizacional es el corazón del sistema SNISSI. Un ADN bien documentado y mantenido asegura:\n- Precisión de agentes\n- Consistencia en decisiones\n- Aprendizaje organizacional\n- Mejora continua\n- Trazabilidad completa\n
